# Интервью по выявлению требований: HTTP API для результатов сегментации

**Дата:** 18.07.2026
**Интервьюер (роль QA/BA):** Claude
**Респондент (роль Research Lead):** владелец проекта

## Заметки по интервью

**В: Кто основные пользователи API с результатами сегментации?**
О: ML-инженеры / исследователи. Им нужен программный доступ к метрикам
по каждому участнику и к агрегированной статистике — без необходимости
открывать Octave или вручную парсить CSV. Например, чтобы подключить
результаты к собственным аналитическим ноутбукам или отслеживать
прогресс пакетной обработки в реальном времени.

**В: Какая главная задача, которую должен решить API?**
О: Обе задачи одинаково важны:
- быстрый просмотр результата по конкретному участнику (отладка
  конкретного случая, проверка, почему Dice Score выглядит подозрительно)
- агрегированная статистика по всей выборке (отслеживание общего
  качества пайплайна, сравнение версий алгоритма во времени)

**В: Насколько критичны права доступа (только чтение vs чтение и запись)?**
О: Возможность записи в будущем понадобится — например, чтобы ревьюер мог
пометить автоматическую маску участника как "требует повторной проверки",
или чтобы удалённо запускать новую пакетную обработку. В текущей же
итерации API сознательно сделан только для чтения (см. роль `qa_readonly`
в `db/schema.sql`): пока команда ещё валидирует сам алгоритм сегментации,
набор данных не должен случайно измениться. Возможность записи вынесена
ниже как будущее расширение, а не как блокер для этой Feature.

## Feature

**Название:** Добавить доступ к результатам автоматической сегментации через HTTP API

**Описание:**
Сейчас у исследователей и ML-инженеров нет способа получить результаты
сегментации, кроме как вручную открыть `results/batch_results.csv` или
запустить скрипты Octave. Эта Feature добавляет небольшой read-only
HTTP API, который отдаёт метрики по каждому участнику и агрегированную
статистику по всей выборке, на основе PostgreSQL вместо плоского CSV-файла.

**Цель / бизнес-ценность:**
Сделать результаты сегментации доступными для запросов и машиночитаемыми,
без необходимости в GNU Octave — чтобы результаты можно было подключать
к другим инструментам (аналитические ноутбуки, дашборды, CI-проверки),
а качество отслеживать без ручного просмотра файлов.

**Критерии успеха:**
- ML-инженер может получить Dice Score и объёмы очага для любого
  обработанного участника одним HTTP-запросом.
- ML-инженер может получить агрегированную статистику (средний/мин/макс
  Dice Score, количество участников) по всей выборке одним HTTP-запросом.
- Ответы API задокументированы и доступны для просмотра через Swagger (`/docs`).
- Исходный набор данных нельзя изменить через API (только чтение —
  осознанное решение для текущей итерации).

**Вне рамок (для текущей итерации):**
- Сам алгоритм сегментации (`src/*.m`) не изменяется.
- Эндпоинты записи/обновления (например, пометка результата, запуск
  повторной обработки) — см. "Возможные доработки" ниже.
- Аутентификация/авторизация сверх уровня роли БД `qa_readonly`.

**Возможные доработки:**
- Эндпоинт с возможностью записи (например,
  `PATCH /results/{subject_code}/review_status`) для пометки участников
  ревьюерами — когда алгоритм станет достаточно стабильным, чтобы риск
  случайной потери данных был меньше, чем на текущем этапе валидации.
- Потребует второй, доступной для записи роли БД в дополнение к
  `qa_readonly`, с собственным журналом изменений (audit trail).

**Статус реализации:** выполнено — см. `api/`, `db/schema.sql`,
`docker-compose.yml`.
