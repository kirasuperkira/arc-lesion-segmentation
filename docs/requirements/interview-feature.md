# Интервью по выявлению требований: HTTP API для результатов сегментации

**Кто основные пользователи API с результатами сегментации?**
ML-инженеры, исследователи, которым необходим программный доступ к метрикам
по каждому участнику и к агрегированной статистике без необходимости запускать
Octave или вручную парсить CSV. Например, чтобы подключить
результаты к собственным аналитическим системам или отслеживать
прогресс пакетной обработки в реальном времени.

**Какая главная задача, которую должен решить API?**
Выделены две ключевые задачи:
- Быстрый просмотр результата по конкретному участнику (отладка
  конкретного случая, проверка);
- Агрегированная статистика по всей выборке (отслеживание общего
  качества пайплайна, сравнение версий алгоритма во времени).

**Насколько критичны права доступа (только чтение/чтение и запись)?**
Возможность записи в будущем понадобится, чтобы
пометить автоматическую маску участника как "требует повторной проверки",
или чтобы удаленно запускать новую пакетную обработку. В текущей
итерации API сознательно сделан только для чтения (роль `qa_readonly`
в `db/schema.sql`): алгоритм сегментации находится в процессе валидации, поэтому
набор данных не должен случайно измениться.

**Как именно будет меняться алгоритм сегментации, и нужно ли это учитывать в структуре API/БД?**
Текущий Dice Score (0.17 ± 0.12) считается критически низким, поэтому план доработки алгоритма
следующий:
- **Этап 0: Диагностика.** Проверка, не является ли низкий Dice
  следствием рассинхронизации пространств (affine/orientation) между
  автоматической и ручной маской, а не слабостью самого порога№
- **Этап 1: Быстрые улучшения порога.** Нормализация интенсивности
  по сканам, bias field correction, подбор параметров постобработки,
  замена фиксированного 95-го перцентиля на Otsu/GMM;
- **Этап 2: Ограничение пространства поиска.** Регистрация в общее
  пространство, исключение желудочков/CSF через атлас, использование
  FLAIR при наличии;
- **Этап 3: ML-модель.** Lesion probability atlas или модель (логрегрессия/random
  forest) поверх порогового подхода.

Каждый этап дает отдельный прогон `batch_evaluate` с собственными
метриками. Важно, чтобы результаты разных версий алгоритма не
перезаписывали друг друга и были сравнимы через API.

**API должен поддерживать версионирование прогонов уже в первой итерации?**
Да, это уточнение к Feature. Сам алгоритм (`src/*.m`) не меняется в рамках этой Feature, но схема
данных должна с самого начала различать результаты по прогонам
(например, `run_id` с привязкой к версии алгоритма плана), а не
хранить единственный плоский набор строк на участника.

## Feature

**Название:** Добавить доступ к результатам автоматической сегментации через HTTP API

**Описание:**
Сейчас у исследователей и ML-инженеров нет способа получить результаты
сегментации, кроме как вручную открыть `results/batch_results.csv` или
запустить скрипты Octave. Эта Feature добавляет read-only
HTTP API, который отдает метрики по каждому участнику и агрегированную
статистику по всей выборке, на основе PostgreSQL вместо плоского
CSV-файла. Схема данных изначально проектируется с учетом того, что
алгоритм сегментации будет проходить через несколько итераций
доработки, поэтому результаты хранятся с привязкой к конкретному прогону.

**Цель:**
Сделать результаты сегментации доступными для запросов и машиночитаемыми,
без необходимости в GNU Octave, чтобы результаты можно было подключать
к другим инструментам (аналитические системы, дашборды, CI-проверки),
а качество отслеживать без ручного просмотра файлов. В том числе
отслеживать, как Dice Score меняется по мере прохождения плана
доработки алгоритма.

**Критерии успеха:**
- ML-инженер может получить Dice Score и объемы очага для любого
  обработанного участника одним HTTP-запросом;
- ML-инженер может получить агрегированную статистику (average/min/max
  Dice Score, количество участников) по всей выборке одним HTTP-запросом,
  с возможностью отфильтровать по конкретной версии алгоритма;
- Результаты разных прогонов хранятся раздельно и сравнимы между собой через API;
- Ответы API задокументированы и доступны для просмотра через Swagger (`/docs`);
- Исходный набор данных нельзя изменить через API.
