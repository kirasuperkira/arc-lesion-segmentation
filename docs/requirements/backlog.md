# Backlog: HTTP API для результатов сегментации

### PBI-1: Просмотр метрик сегментации одного участника

**Как** ML-инженер,
**я хочу** получить Dice Score и объемы очага для конкретного участника
по его коду, с возможностью указать конкретный прогон алгоритма,
**чтобы** быстро посмотреть или отладить один результат, в том числе
сравнить, как менялся результат по этому участнику между этапами
доработки алгоритма.

**Критерии приемки:**
- **Дано:** участник с известным `subject_code` существует в базе данных,
  **когда** отправляю `GET /results/{subject_code}` без `run_id`,
  **тогда** ответ `200 OK` и содержит `dice_score`, `lesion_volume_gt`,
  `lesion_volume_auto`, `run_id` — по последнему прогону (`is_latest = true`).
- **Дано:** участник обработан в нескольких прогонах,
  **когда** отправляю `GET /results/{subject_code}?run_id={id}`,
  **тогда** ответ содержит метрики именно этого прогона, а не последнего.
- **Дано:** указан `run_id`, которого не существует,
  **когда** отправляю такой запрос,
  **тогда** ответ `404 Not Found` с сообщением, где указан отсутствующий `run_id`.
- **Дано:** `subject_code`, которого не существует,
  **когда** отправляю `GET /results/{subject_code}`,
  **тогда** ответ `404 Not Found` с сообщением об ошибке, где указан
  отсутствующий код участника.
- **Дано:** эндпоинт задокументирован,
  **когда** открываю `/docs`,
  **тогда** `GET /results/{subject_code}` присутствует в списке со схемой
  ответа, включая параметр `run_id` и поле `run_id` в ответе.

### PBI-2: Просмотр агрегированной статистики по выборке

**Как** ML-инженер,
**я хочу** получить average/min/max Dice Score и общее количество
обработанных участников для конкретного прогона алгоритма,
**чтобы** отслеживать качество пайплайна и сравнивать его между
итерациями доработки.

**Критерии приемки:**
- **Дано:** в базе данных есть обработанные участники в нескольких
  прогонах,
  **когда** отправляю `GET /stats` без `run_id`,
  **тогда** ответ `200 OK` и содержит `participants_count`, `avg_dice`,
  `min_dice`, `max_dice`, `run_id` рассчитанные только по последнему
  прогону.
- **Дано:** конкретный `run_id`, соответствующий исходному baseline-прогону,
  **когда** отправляю `GET /stats?run_id={id}`,
  **тогда** `avg_dice` совпадает со средним значением из
  `results/batch_results.csv` (0.1704) с точностью до округления.
- **Дано:** два разных `run_id` (например, baseline и прогон после Этапа 1),
  **когда** запрашиваю `/stats` для каждого,
  **тогда** значения `avg_dice` различаются и корректно отражают только
  участников соответствующего прогона (не смешиваются между прогонами).

### PBI-3: Постраничный просмотр списка обработанных участников

**Как** ML-инженер,
**я хочу** получать список участников с пагинацией, отфильтрованный по
конкретному прогону,
**чтобы** просматривать результаты постранично в рамках одной версии
алгоритма, не загружая весь набор данных целиком и не путая результаты
разных прогонов.

**Критерии приемки:**
- **Дано:** участников больше, чем запрошенный `limit`,
  **когда** отправляю `GET /results?limit=5`,
  **тогда** ответ содержит не более 5 элементов, все — из последнего прогона.
- **Дано:** указан `run_id`,
  **когда** отправляю `GET /results?run_id={id}&limit=5`,
  **тогда** возвращаются только участники этого прогона.
- **Дано:** некорректный `limit` (0, отрицательный, нечисловой или выше
  максимума в 500),
  **когда** отправляю такой запрос,
  **тогда** ответ `422 Unprocessable Entity`.

### PBI-6: Просмотр списка прогонов алгоритма

**Как** ML-инженер или Research Lead,
**я хочу** получить список всех прогонов алгоритма с их метаданными
(версия/этап алгоритма, дата, признак последнего),
**чтобы** понимать, какие `run_id` доступны для сравнения, не заглядывая
напрямую в базу данных.

**Критерии приемки:**
- **Дано:** в базе данных есть несколько прогонов,
  **когда** отправляю `GET /runs`,
  **тогда** ответ `200 OK` содержит список объектов с `run_id`, `label`
  (например, "baseline", "stage-1-threshold-fix"), `created_at`, `is_latest`.
- **Дано:** ни один прогон не обработан полностью (в процессе),
  **когда** запрашиваю `GET /runs`,
  **тогда** такой прогон помечен `status: "in_progress"` и не участвует
  в `is_latest`, пока не завершится.
- **Дано:** эндпоинт задокументирован,
  **когда** открываю `/docs`,
  **тогда** `GET /runs` присутствует в списке со схемой ответа.

### PBI-4: Ограничение доступа API только на чтение

**Как** QA-инженер, отвечающий за целостность данных,
**я хочу**, чтобы API подключался к базе данных с ролью, у которой есть
только права `SELECT`,
**чтобы** ошибка в коде API никогда не могла случайно изменить или удалить
результаты сегментации, пока алгоритм в процессе валидации.

**Критерии приемки:**
- **Дано:** роль `qa_readonly`, определенная в `db/schema.sql`,
  **когда** проверяю права,
  **тогда** у есть только `SELECT`; `INSERT`, `UPDATE`, `DELETE` ни на одной таблице, включая новую таблицу `runs`.
- **Дано:** API настроен на подключение под ролью `qa_readonly`,
  **когда** через это подключение предпринимается любая попытка записи
  в базу данных,
  **тогда** PostgreSQL отклоняет с ошибкой прав доступа.

### PBI-5: Восстановление набора данных из резервной копии

**Как** QA/DevOps-инженер,
**я хочу** иметь задокументированные скрипты `backup.sh` / `restore.sh`,
**чтобы** можно было восстановить набор результатов сегментации (включая
историю прогонов) после случайной потери данных или неудачной миграции.

**Критерии приемки:**
- **Дано:** заполненная база данных с несколькими прогонами,
  **когда** запускаю `db/backup.sh`,
  **тогда** создается файл `.dump` с меткой времени в `db/backups/`,
  включающий таблицы `participants`, `results` и `runs`.
- **Дано:** файл резервной копии и пустая или повреждённая база данных,
  **когда** запускаю `db/restore.sh <файл>`,
  **тогда** база данных восстанавливается, и `SELECT COUNT(*) FROM runs`
  возвращает исходное количество прогонов.
